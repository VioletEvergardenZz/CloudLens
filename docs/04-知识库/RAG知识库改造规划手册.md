# RAG 知识库改造规划手册

> 文档状态：Draft  
> 适用范围：`go-watch-file/`、`console-frontend/`、`docs/04-知识库/`  
> 编写日期：2026-03-19  
> 目标边界：本轮只做 A 类任务范围内的知识库闭环增强，不扩展到多租户、复杂工作流 DSL、完整 RBAC 或大规模模型治理平台

## 1. 本轮目标

把当前“知识库沉淀、检索与引用式问答”从“文章级检索 + 文章级问答拼装”升级为“可切片检索、可引用回溯、可降级运行”的 RAG 知识库。

本轮需要同时满足以下目标：

1. 保留现有知识条目生命周期能力：草稿、审核、发布、归档、回滚不能退化。
2. 保留现有控制台主入口和 `/api/kb/*` 路由骨架，避免前后端大面积推翻。
3. 把检索粒度从“整篇文章”下沉到“知识片段（chunk）”。
4. 把问答上下文从“整篇正文拼接”升级为“TopK 片段拼接 + 引用回传”。
5. AI 不可用时仍能返回可追溯的本地回答，保持降级可用。
6. 测试门禁、指标口径和文档说明保持同步。

## 2. 现状代码盘点

### 2.1 后端现状

当前知识库主链路主要落在以下文件：

1. `go-watch-file/internal/kb/service.go`
2. `go-watch-file/internal/kb/types.go`
3. `go-watch-file/internal/api/kb_handlers.go`
4. `go-watch-file/internal/metrics/prometheus.go`
5. `go-watch-file/cmd/kb-eval/main.go`

现状能力已经比较完整：

1. SQLite 持久化、版本记录、审核流、回滚、Markdown 导入已经具备。
2. `/api/kb/search`、`/api/kb/ask`、`/api/kb/recommendations` 已可用。
3. `/api/kb/ask` 已支持“检索后可选 AI 增强”，并具备 `degraded/errorClass/fallbackReason` 降级元信息。
4. `kb-eval hitrate/citation/mttd`、`/metrics`、`/api/kb/gates` 已经形成了质量门禁骨架。

### 2.2 现状不足

当前实现离真正可维护的 RAG 还差几层关键能力：

1. 检索粒度仍是文章级。
   说明：`service.go` 中 `Search` 仍基于 `ListArticles + LIKE + 分词兜底`，缺少片段索引。

2. 问答上下文仍偏粗。
   说明：`kb_handlers.go` 中虽然存在 `ragPayload` 命名，但实际输入模型的仍是文章摘要和正文截断，不是片段级上下文。

3. 引用粒度仍偏粗。
   说明：当前 `Citation` 只到 `articleId/title/version`，还缺少“命中的是哪一段、哪一节、哪段摘要”的证据面。

4. 导入链路没有建立 RAG 索引。
   说明：`ImportDocs` 只是把 Markdown 导入为文章版本，没有同步生成片段索引。

5. 存量数据没有再索引机制。
   说明：数据库迁移后，历史条目若不重新切片，将无法被新检索链路稳定命中。

6. 前端已经写着“语义检索”，但后端并没有真正的语义或片段检索底座。
   说明：这会造成术语和行为不一致，后续必须统一口径。

## 3. 改造原则

本轮遵循以下强约束：

1. 最小改动优先。
   只增强知识库内部实现，不推翻控制台结构、不拆微服务。

2. 兼容接口优先。
   现有 `/api/kb/articles`、`/api/kb/search`、`/api/kb/ask`、`/api/kb/recommendations` 保持可用。

3. 数据可追溯优先。
   任何新引入的片段索引都必须与 `article_id + version` 绑定，避免“问答引用无法回放”。

4. 降级可用优先。
   即使外部 AI 不可用，检索与引用式回答仍然要成立。

5. 分阶段演进。
   本轮先做“片段化 RAG 基座”，向量检索与混合召回作为下一阶段增强，不在本轮强行塞入。

## 4. 本轮 RAG 定义

本项目本轮将 RAG 定义为：

1. 对知识正文按规则切片，生成可检索的 chunk。
2. 基于问题先召回 chunk，而不是直接召回整篇文章。
3. 基于 TopK chunk 组织回答上下文。
4. 返回回答时附带可追溯引用，至少能定位到文章、版本、片段摘要。
5. 模型不可用时，仍能基于 TopK chunk 返回本地降级回答。

说明：

1. 本轮不强依赖外部 embedding 服务。
2. 本轮允许采用“片段检索 + 规则重排 + 生成增强”的实现方式。
3. 下一阶段若接入 embedding，可在不破坏现有 API 的前提下叠加混合召回。

## 5. 目标架构

### 5.1 保留层

以下能力继续保留并复用：

1. `kb_articles`：文章元信息与生命周期状态。
2. `kb_article_versions`：正文版本与来源记录。
3. `kb_reviews`：审核与回滚轨迹。
4. `kb_references`：来源引用。
5. 前端知识库控制台：条目列表、编辑、审核、回滚、问答、推荐入口。

### 5.2 新增层

新增一层“RAG 索引层”，放在知识库服务内部：

1. 片段切分器
   负责把 Markdown 正文切成多个 chunk，并尽量保留标题上下文。

2. 片段索引表
   把当前版本正文映射为可检索的 chunk 记录。

3. 片段召回器
   基于问题召回 TopK chunk，并输出命中分数、命中摘要。

4. 问答上下文组装器
   把召回片段组织为模型提示词或本地降级回答材料。

5. 引用回传器
   把 chunk 命中结果折叠为用户可理解的引用信息。

### 5.3 架构示意

```text
知识条目录入/导入
    -> 写入 article/version
    -> 生成 chunk 索引
    -> 检索时召回 chunk
    -> 问答时拼装 TopK chunk
    -> 返回 answer + citations + degraded/meta
```

## 6. 数据模型设计

### 6.1 保留现有表

继续保留：

1. `kb_articles`
2. `kb_article_versions`
3. `kb_tags`
4. `kb_article_tags`
5. `kb_reviews`
6. `kb_references`

### 6.2 新增表

新增 `kb_article_chunks`：

1. `id`
   片段主键。

2. `article_id`
   对应文章 ID。

3. `version`
   对应文章版本。

4. `chunk_index`
   同一版本内的顺序号。

5. `heading`
   片段所属章节标题，便于回答时给出上下文。

6. `content`
   片段正文。

7. `chunk_hash`
   用于追踪片段内容与排查重复。

8. `created_at`
   片段生成时间。

说明：

1. 片段必须绑定到 `article_id + version`，不能只绑定文章，否则历史版本无法回放。
2. 本轮不单独新增 embedding 表，避免在没有稳定外部 embedding 服务前引入脆弱依赖。

### 6.3 后续预留

下一阶段若接入向量能力，再新增：

1. `kb_chunk_embeddings`
2. `embedding_model`
3. `embedding_dim`
4. `vector_payload`
5. `updated_at`

本轮规划文档先保留接口与命名空间，不在当前开发中强行落地。

## 7. 检索链路设计

### 7.1 检索目标

从“找文章”升级为“先找最相关片段，再折叠为文章结果”。

### 7.2 召回流程

1. 规范化查询词。
2. 生成问题候选词和分词 token。
3. 在当前可见版本的 `kb_article_chunks` 中做片段级召回。
4. 根据标题、摘要、片段标题、片段正文做综合打分。
5. 取每篇文章分数最高的 chunk，折叠为搜索结果。
6. 把最佳 chunk 摘要挂到结果上，供前端展示命中原因。

### 7.3 排序原则

建议权重：

1. 标题命中权重最高。
2. 摘要与章节标题次之。
3. 片段正文命中再次之。
4. 精确短语命中高于散列 token 命中。
5. 同分时优先最近更新的条目。

### 7.4 搜索接口兼容策略

`POST /api/kb/search` 保持原路径不变，但返回值增强为：

1. `items`
   仍然返回文章列表，兼容前端当前结构。

2. `items[].matchSnippet`
   新增，可选，表示最佳命中片段摘要。

3. `items[].matchScore`
   新增，可选，便于前端或排障观察。

4. `items[].content`
   搜索场景下可返回命中的片段正文，而不是整篇正文，避免无关上下文过大。

## 8. 问答链路设计

### 8.1 目标

把当前“文章级问答”升级为“TopK chunk 上下文问答”。

### 8.2 流程

1. 接收问题。
2. 生成问题候选词。
3. 召回 TopK chunk。
4. 生成引用列表。
5. 若 AI 可用：
   使用 TopK chunk 作为上下文调用模型，输出结构化 `answer/confidence`。
6. 若 AI 不可用或失败：
   使用最高分 chunk 生成本地降级回答。

### 8.3 引用策略

当前引用最少包含：

1. `articleId`
2. `title`
3. `version`
4. `snippet`
5. `chunkIndex`
6. `heading`

兼容性要求：

1. 老前端即使只读取 `articleId/title/version` 也不会出错。
2. 新前端可以进一步展示引用片段和命中章节。

### 8.4 Prompt 约束

模型侧必须遵守：

1. 只能基于提供的 chunk 回答。
2. 不允许编造未提供的排障步骤。
3. 输出保持结构化 JSON。
4. 最终 citations 仍由系统回填，不依赖模型自由生成。

## 9. 导入与沉淀链路设计

### 9.1 创建/更新/回滚

以下动作都必须同步重建当前版本 chunk：

1. `CreateArticle`
2. `UpdateArticle`
3. `RollbackArticle`
4. `ImportDocs`

### 9.2 存量数据补齐

数据库迁移完成后必须执行一次“当前版本重建索引”：

1. 扫描现有文章当前版本。
2. 如果当前版本没有 chunk，则补建。
3. 该过程应在服务启动时自动完成，避免手工漏操作。

### 9.3 导入口径

`/api/kb/import/docs` 继续保留：

1. 仍按来源路径去重与增量更新。
2. 但导入成功后必须自动生成片段索引。
3. 导入失败不能污染已有索引。

## 10. 前端改造设计

### 10.1 知识库控制台

本轮前端只做最小增强：

1. 把“语义检索”文案改为更准确的“RAG 检索”或“片段检索”。
2. 在搜索结果中展示 `matchSnippet`，告诉用户命中了哪一段。
3. 在问答结果中展示更细粒度引用信息，例如章节或片段摘要。
4. 保留当前 `degraded/errorClass/fallbackReason` 展示，不去掉降级可视化。

### 10.2 告警推荐

`/api/kb/recommendations` 本轮继续返回文章列表，但底层排序改为 chunk 召回结果折叠后的文章排序。这样告警详情页不需要大改就能获得更准确的推荐结果。

## 11. 指标与验证设计

### 11.1 继续保留的指标

继续使用现有指标：

1. `gwf_kb_search_total`
2. `gwf_kb_search_hit_ratio`
3. `gwf_kb_ask_total`
4. `gwf_kb_ask_citation_ratio`
5. `gwf_kb_review_latency_ms`

### 11.2 本轮建议增加的观测点

若改动成本可控，建议追加内部日志或后续指标：

1. chunk 总数
2. 每篇文章平均 chunk 数
3. 问答使用的 TopK chunk 数
4. 降级回答占比

说明：

1. 本轮以先完成 RAG 主链路为主。
2. 若时间不足，可先不暴露新的 Prometheus 指标，只保留代码内部可排查信息。

## 12. 兼容性与回滚策略

### 12.1 API 兼容

以下接口路径不变：

1. `GET /api/kb/articles`
2. `POST /api/kb/search`
3. `POST /api/kb/ask`
4. `POST /api/kb/import/docs`
5. `GET /api/kb/recommendations`

### 12.2 数据回滚

本轮新增表是附属索引表，回滚策略如下：

1. 如果 RAG 检索出现问题，可退回到旧的文章级搜索逻辑。
2. `kb_article_chunks` 可删除并重建，不影响主数据表。
3. 主数据依旧以 `kb_articles`、`kb_article_versions` 为准。

### 12.3 运行时降级

问答链路必须支持三层兜底：

1. 最优：RAG 检索 + AI 生成
2. 中间：RAG 检索 + 本地模板回答
3. 最差：返回“未找到可引用条目”，但不能返回空引用的伪答案

## 13. 实施拆解

### 13.1 第一阶段：RAG 基座

本轮实际开发优先做以下内容：

1. 新增 chunk 数据表与迁移。
2. 新增 Markdown 切片逻辑。
3. 在创建、更新、回滚、导入时同步构建 chunk。
4. 新增存量条目当前版本补索引。
5. 将 `Search` 升级为片段召回并折叠文章。
6. 将 `Ask` 升级为基于 TopK chunk 的回答与引用。
7. 前端展示 `matchSnippet` 与更细粒度引用信息。

### 13.2 第二阶段：混合召回增强

下一阶段再评估：

1. 外部 embedding 服务接入
2. chunk 向量缓存
3. 词法召回 + 向量召回混排
4. 召回质量离线评估集扩充

### 13.3 第三阶段：运营与治理

后续可扩展：

1. 片段质量门禁
2. 文档模板规范化

## 14. 本轮验收门禁

必须通过：

1. `cd go-watch-file && go test ./... -count=1`
2. `cd console-frontend && npm run build`

建议补充验证：

1. 导入 Markdown 后能生成 RAG 检索结果。
2. `POST /api/kb/search` 返回的结果带有命中片段。
3. `POST /api/kb/ask` 在 AI 关闭时仍能返回可引用回答。
4. 告警推荐仍可正常显示。
5. `kb-eval hitrate` 与 `kb-eval citation` 口径不退化。

## 15. 风险与取舍

### 15.1 为什么本轮不直接上向量数据库

原因：

1. 当前仓库是单机 SQLite 架构，强上外部向量库会显著放大部署和维护成本。
2. 现有配置体系只有通用 AI 配置，没有稳定 embedding 配置面。
3. 本阶段目标强调“可观测、可回归、可追溯收口”，优先级高于技术堆栈升级。

### 15.2 为什么不推翻现有 `/api/kb/*`

原因：

1. 前端、验收脚本、`kb-eval`、阶段复盘脚本都已经依赖当前接口。
2. 推翻接口会把工作从“增强知识库”变成“重做一套系统”，不符合最小改动原则。

### 15.3 本轮已知限制

1. 本轮的 RAG 先以片段检索为主，不承诺真正的 embedding 语义检索。
2. 片段切分质量会影响最终效果，需要结合真实文档持续调优。
3. `kb-eval` 的样本可能需要后续扩充，避免只覆盖旧式关键词场景。

本轮预计产出：

1. 后端 RAG 基座代码
2. 前端知识库最小增强
3. 中文文档同步

## 17. 结论

本次改造不应理解为“把现有知识库推倒重来”，而应理解为：

1. 保留文章生命周期与控制台骨架。
2. 在服务内部新增片段化 RAG 检索层。
3. 用兼容方式把检索、问答、推荐升级为更像真正 RAG 的实现。

这样能在不扩边界的前提下，把“知识沉淀 -> 检索 -> 引用式问答 -> 告警推荐”的闭环，从“能用”升级到“更可解释、更可维护、更接近生产”。
