# 统一运维平台产品方向收敛与开源对标（2026-04-20）

> 文档状态：设计收敛稿  
> 适用范围：`console-frontend/` 原型收敛、统一运维控制台信息架构优化、后续 A 类能力落地  
> 目标边界：不扩张范围，围绕 `文件入云 + 告警决策 + AI 分析 + 知识库复用` 的事件闭环做产品收敛

## 1. 这次收敛要解决什么

前几轮设计已经把方向从“模块控制台集合”推向了“统一运维工作台”，但仍然有一个核心问题没有完全解决：

`用户进来以后，第一眼到底该先看什么，第二步该去哪里，第三步在哪个页面收口。`

当前不满意感主要来自 3 个点：

1. 顶层导航虽然已经改成了工作域，但 6 个域仍然几乎等权，主次还不够明确。
2. 首页仍然偏“平台总览”，还不是“值班入口”。
3. 事件页、系统页、接入页、执行页之间的角色边界还不够锋利，用户仍然容易靠自己脑补流程。

所以这次不再继续“补页面”，而是直接收敛成下面这个产品判断：

`GWF 的核心不是全栈监控平台，也不是 CMDB，也不是自动化平台，而是一套面向值班与事件闭环的统一运维工作台。`

更具体地说：

- 主对象是 `事件`
- 主上下文是 `系统`
- 主入口是 `值班驾驶舱`
- 接入、知识、执行是为这条主线服务的支撑域

## 2. GitHub 开源方案对标结论

本次对标重点不是找一个“跟 GWF 一模一样”的产品，而是拆成 4 类代表方案，分别看它们最擅长解决什么。

### 2.1 可观测平台

- [SigNoz](https://github.com/SigNoz/signoz)
- [OpenObserve](https://github.com/openobserve/openobserve)

可借鉴点：

- 把日志、指标、链路、告警放到统一上下文里，而不是让用户自己跨模块拼证据。
- 首页不是堆卡片，而是快速回答“哪里出问题了、现在最危险的是什么”。
- 告警不是孤立对象，而是和证据、查询、趋势一起出现。

不适合直接照搬的点：

- GWF 当前阶段不需要做完整 observability 平台，不应该被日志查询器、指标工作台、报表系统牵着走。

### 2.2 事件响应 / 值班 / 状态页

- [OneUptime](https://github.com/OneUptime/oneuptime)
- [GoAlert](https://github.com/target/goalert)
- [Uptime Kuma](https://github.com/louislam/uptime-kuma)

可借鉴点：

- 值班产品都非常强调 `责任人、升级链、当前状态、下一动作`。
- 用户打开系统后，最先看到的是事件压力，不是配置项列表。
- “谁来处理、是否超时、有没有 SOP” 这些信息必须站在右手边或第一屏。

不适合直接照搬的点：

- GWF 不需要做完整 PagerDuty 替代，也不需要先把 on-call 排班做成主产品。
- 状态页能力可以是结果输出，但不应该反过来定义整个平台结构。

### 2.3 系统目录 / 软件目录 / 事实源

- [NetBox](https://github.com/netbox-community/netbox)
- [Backstage](https://github.com/backstage/backstage)

可借鉴点：

- 目录页的价值不在“列全资产”，而在提供稳定的 `source of truth`。
- 单系统详情必须同时包含：负责人、环境、服务、入口、健康、文档、最近变化、最近事件。
- 目录对象要能成为事件、接入、执行、知识的连接点。

不适合直接照搬的点：

- GWF 当前阶段不做重型 CMDB，也不做大而全开发者门户。
- 系统目录应该服务事件闭环，而不是单独扩张成独立产品线。

### 2.4 执行 / Runbook Automation

- [Rundeck](https://github.com/rundeck/rundeck)

可借鉴点：

- 动作中心的核心不是“任务很多”，而是 `动作可复用、结果可追踪、失败可审计`。
- 自动化不应该漂浮在外部，而应该回链到事件与系统。

不适合直接照搬的点：

- GWF 当前阶段不做复杂工作流 DSL，也不做超重编排系统。

## 3. 对 GWF 的产品收敛判断

综合对标后，GWF 更适合定义为：

> 一套面向值班工程师 / SRE / 平台运维的事件闭环工作台。  
> 它以值班入口承接风险判断，以事件工作台收口诊断与动作，以系统详情承接上下文，并用接入、知识、执行三类支撑能力补齐闭环。

这意味着顶层要从“6 个域并列”进一步收敛成：

### 3.1 三个主线域

1. 值班
2. 事件
3. 系统

### 3.2 三个支撑域

1. 接入
2. 知识
3. 执行

换句话说：

- `值班` 负责回答“现在先处理什么”
- `事件` 负责回答“这件事怎么判断、怎么处置、怎么留痕”
- `系统` 负责回答“这件事到底挂在哪个系统与服务上”
- `接入` 负责补齐输入和纳管事实
- `知识` 负责补齐 SOP 与经验沉淀
- `执行` 负责让动作真正落地，并把结果审计回链

## 4. 本次前端原型收敛结果

### 4.1 值班驾驶舱

首页不再只是“平台总览”，而是改为：

- 当前优先队列
- 交接与判断
- 系统热点
- 闭环主线观察
- 快速进入事件 / 系统 / 接入 / 执行

核心变化：

- 先告诉值班人现在最危险的是什么
- 再告诉值班人为什么先做这个
- 最后给出下一步入口

### 4.2 事件工作台

事件页不再只是告警摘要，而是收口为单事件工作台：

- 左侧：事件队列与时间线
- 中间：诊断面，放证据、判断、影响面
- 右侧：动作面，放 owner、SOP、任务和跳转入口
- 下方：系统上下文与复盘草稿

核心变化：

- 不要求用户自己在告警页、控制面、知识库之间来回切换
- 让“判断”和“动作”同时存在于同一屏

### 4.3 系统台账 / 系统详情

系统页从“表格式台账”继续收敛成：

- 系统目录筛选
- 单系统详情
- 环境与服务清单
- 最近事件
- SOP / Runbook
- 待补齐项

核心变化：

- 系统对象成为事件页的上下文承接面
- 不再只展示环境和服务，还必须展示责任、文档、事件与 SOP

### 4.4 导航主次

侧边导航不再把 6 个工作域完全等权展示，而是显式分为：

- 主线工作域：值班 / 事件 / 系统
- 支撑工作域：接入 / 知识 / 执行

这一步非常重要，因为它直接决定用户会不会在第一屏迷路。

## 5. 为什么这套方向更符合当前阶段

它更符合当前阶段目标，因为它没有扩边界，而是把已有能力重新组织：

- 文件入云继续放在接入域，但不再冒充首页中心
- 告警决策与 AI 分析继续存在，但被收口到事件工作台
- 知识库继续存在，但从“文档区”转成“沉淀与 SOP 复用中心”
- 控制面继续存在，但从“独立功能页”转成事件动作支撑面

也就是说，这不是功能扩张，而是结构收口。

## 6. 后续建议

如果沿着这条路线继续推进，下一步建议优先做：

1. 把事件对象和系统对象的最小数据契约统一下来。
2. 让系统详情页真正能联动最近事件、健康规则和 SOP。
3. 让事件工作台支持最小动作回写，例如认领、备注、生成 SOP 草稿。
4. 让接入草案可以进入系统详情，而不是停在原型页。

## 7. 一句话结论

这次不再把 GWF 继续往“大而全运维平台”推，而是把它收敛成一套：

`值班先判断、事件页收口、系统页承接上下文、支撑域围绕主线服务` 的统一运维工作台。
