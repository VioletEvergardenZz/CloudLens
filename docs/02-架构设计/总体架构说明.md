# 总体架构

```text
CloudLens Frontend
    ↓
CloudLens Backend
    ├─ Cloud Accounts
    ├─ Unified Resource Index
    ├─ Aliyun ECS/RDS ReadOnly API
    ├─ Huawei ECS/RDS/CES ReadOnly API
    ├─ Kubernetes ReadOnly Inspection
    ├─ Alert / AI / KB
    └─ File Upload Extension
    ↓
SQLite + Aliyun OpenAPI + Huawei Cloud SDK + Kubernetes API + Optional OSS
```

## 目标架构分层

CloudLens 后续按“接口层、应用服务层、适配器层、存储层”收口：

```text
React 控制台
    ↓
HTTP API
    ↓
应用服务层
    ├─ 云账号服务
    ├─ 云资源服务
    ├─ 指标服务
    ├─ 巡检风险服务
    └─ K8s 只读巡检服务
    ↓
适配器层
    ├─ 阿里云 ECS/RDS/CMS
    ├─ 华为云 ECS/RDS/CES
    └─ Kubernetes client-go
    ↓
SQLite 快照与索引
```

## 后端代码组织

后端当前采用轻量 Go 项目结构，不引入大型 Web 框架。HTTP 层使用标准库 `net/http` handler 风格，并引入 `chi` 只做路由分组，让接口地图更清晰：

```text
cmd/main.go
    ↓
internal/api/server.go
    ↓
internal/api/routes.go
    ↓
internal/api/*_handlers.go
    ↓
internal/app/
    ↓
internal/cloud/ + internal/store/
```

关键目录职责：

- `internal/api`：HTTP 接口层，`routes.go` 是路由地图，handler 只做参数校验、错误映射和响应组装。
- `internal/app`：应用服务层，承接跨存储、跨云厂商的业务编排。
- `internal/cloud`：云厂商只读适配器，封装阿里云、华为云 SDK。
- `internal/store`：SQLite 存储。
- `internal/service`：长生命周期运行态服务，例如文件监听、上传队列和告警热更新。

当前已经落地的纵切：

- `internal/app/cloud`：沉淀统一资源、巡检风险和 K8s Node 关联应用服务。
- `internal/store`：新增 `cloud_resources` 规范化资源索引表，保留原有快照表作为降级来源。
- `/api/cloud/resources` 与 `/api/resources`：从本地 ECS/RDS 快照生成统一资源索引，用于风险、筛选和 K8s Node 关联。
- `/api/resources/{id}` 与 `/api/resources/{id}/metrics`：提供统一资源详情和统一指标入口，内部仍复用阿里云/华为云只读适配器。
- `/api/inspection/risks`：基于统一资源索引输出巡检风险，作为后续替换 `/api/cloud/risks` 的新入口。
- `/api/k8s/overview`：基于 `client-go` 读取 Node、Namespace、Pod、Deployment 和 Warning Event，只做只读巡检。
- `/api/k8s/node-links`：按 Node 标签、`providerID`、`InternalIP` 和主机名匹配云主机资源。
- 前端拆出 `src/cloud/pages/K8sInspectionPage.tsx`：展示节点就绪、Pod 状态、工作负载、异常信号、Pod 明细和 Node 关联。

这次没有直接把旧 handler 全量搬到新目录，是为了保留现有云账号、快照降级和前端主线的稳定性。后续迁移时，优先把 `internal/api` 中的云资源编排逐步下沉到应用服务层。

## Kubernetes 部署拓扑

```text
Browser / kubectl port-forward
    ↓
cloudlens-frontend Service
    ↓
cloudlens-frontend Deployment (nginx)
    ↓ /api
cloudlens-backend Service
    ↓
cloudlens-backend Deployment (Go API)
    ├─ ConfigMap: /etc/cloudlens/config.yaml
    ├─ Secret: 云厂商 AK/SK、AI Key
    ├─ PVC: SQLite 数据、日志、config.runtime.yaml
    ├─ Probe: /api/health
    └─ Metrics: /metrics
```

K8s 部署保持当前产品边界：

- 后端使用 SQLite，本阶段固定单副本，避免多个 Pod 同时写入本地数据库。
- 前端可以独立扩展，但默认仍保持 1 副本，避免在学习和演示环境中引入额外复杂度。
- 后端 Service 只暴露集群内访问，默认由前端 nginx 代理 `/api`。
- NetworkPolicy 限制后端入口来源，ServiceAccount 禁止默认 token 自动挂载。
- 如需公网访问，后续再按目标集群补 Ingress 和 TLS，不作为默认清单。
- K8s 只读巡检默认优先读取本地 `kubeconfig`；如需在集群内巡检当前集群，需要显式应用 `deploy/k8s/k8s-readonly-rbac.yaml` 并打开后端 ServiceAccount token 挂载。

当前默认使用本地 SQLite：

- `data/cloud/cloud.db`
- `data/control/control.db`
- `data/kb/knowledge.db`

默认不依赖 OSS，也不依赖独立数据库。

补充说明：

- 当前前后端主线围绕阿里云 ECS/RDS 与华为云 ECS/RDS 监控展开
- 系统资源监控属于旧支线，不再作为当前产品主线继续推进
