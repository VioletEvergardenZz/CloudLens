# 启动与配置

这份文档只保留 CloudLens 当前阶段最重要的启动和配置说明。建议先按“本地开发 -> Docker Compose -> Kubernetes”的顺序理解，不要一开始就从 K8s YAML 入手。

## 先选启动方式

| 场景 | 推荐方式 | 适合谁 |
| --- | --- | --- |
| 改代码、学项目 | 本地开发 | 日常开发和调试 |
| 想快速像产品一样跑起来 | Docker Compose | 单机部署和演示 |
| 已经有集群，要交给运维环境运行 | Kubernetes | 集群部署 |

## 本地开发

后端：

```bash
cd cloudlens-backend
go run ./cmd
```

前端：

```bash
cd cloudlens-frontend
npm ci
npm run dev
```

本地开发需要 Go、Node.js 和 npm。没有云账号也能启动，只是云资源页面会显示空数据或降级信息。

## Docker Compose

```bash
docker compose up --build -d
```

默认访问：

| 服务 | 地址 |
| --- | --- |
| 前端控制台 | `http://localhost:8081` |
| 后端接口 | `http://localhost:8082` |

不接真实云账号时可以直接启动，云资源页面会保持空数据或降级提示。需要接云账号、修改端口或覆盖 CORS 时，再执行 `cp .env.example .env` 并编辑 `.env`。

`.env` 是 Docker Compose 场景最主要的可选配置入口，云账号、端口、CORS 都优先放这里。

默认的 `docker-compose.yml` 使用 Docker 命名卷保存 `/etc/cloudlens`、`/app/logs` 和 `/app/data`，本地演示时不需要提前创建宿主机目录。需要启用 Docker 容器日志告警时，再按需打开 `docker-compose.yml` 中的 `docker.sock` 挂载注释。

## Kubernetes

K8s 不是开发入口，它适合把 CloudLens 放进已有集群中运行。

先构建镜像：

```bash
docker build -t cloudlens-backend:local ./cloudlens-backend
docker build -t cloudlens-frontend:local ./cloudlens-frontend
```

如果用的是 kind 这类本地集群，还需要把镜像加载进集群：

```bash
kind load docker-image cloudlens-backend:local
kind load docker-image cloudlens-frontend:local
```

如果只是想快速部署，不想先看多个 YAML：

```bash
kubectl apply -f deploy/k8s/all-in-one.yaml
```

如果要按模块维护清单：

```bash
kubectl apply -k deploy/k8s
```

访问前端：

```bash
kubectl -n cloudlens port-forward svc/cloudlens-frontend 8081:80
```

验证后端：

```bash
kubectl -n cloudlens port-forward svc/cloudlens-backend 8082:8080
curl http://localhost:8082/api/health
curl http://localhost:8082/api/dashboard?mode=light
curl http://localhost:8082/api/runtime/checks
curl http://localhost:8082/api/cloud/diagnostics
curl http://localhost:8082/api/cloud/risks
curl http://localhost:8082/api/cloud/inspection-report
curl http://localhost:8082/api/resources
curl http://localhost:8082/api/inspection/risks
curl http://localhost:8082/api/k8s/overview
curl http://localhost:8082/api/k8s/node-links
curl http://localhost:8082/metrics
```

## 配置怎么理解

CloudLens 当前有三类配置：

| 配置 | 用在哪里 | 说明 |
| --- | --- | --- |
| `.env` | 本地和 Docker Compose | 最容易理解，适合放环境变量 |
| `config.yaml` | 后端静态配置 | 容器内默认是 `/etc/cloudlens/config.yaml` |
| `config.runtime.yaml` | 控制台保存的运行态配置 | 后端自动读写，不需要手工维护 |

K8s 里对应关系：

| K8s 对象 | CloudLens 里负责什么 |
| --- | --- |
| `ConfigMap` | 普通配置，相当于配置文件 |
| `Secret` | 云厂商 AK/SK、AI Key 等敏感信息 |
| `PVC` | SQLite 数据、日志、运行态配置 |
| `Deployment` | 后端和前端进程 |
| `Service` | 集群内访问入口 |
| `ServiceAccount` | 禁用默认 token 自动挂载，减少 Pod 凭据暴露 |
| `NetworkPolicy` | 限制后端只接收前端和监控命名空间访问 |

Kubernetes 只读巡检默认适合本地 kind 验证。后端会依次读取 `CLOUDLENS_K8S_KUBECONFIG`、`KUBECONFIG` 和 `~/.kube/config`。如果希望部署在集群内的 CloudLens 巡检当前集群，需要额外应用：

```bash
kubectl apply -f deploy/k8s/k8s-readonly-rbac.yaml
```

并显式把后端 Deployment 的 `automountServiceAccountToken` 调整为 `true`。这个动作会让后端 Pod 获得 Kubernetes API 只读凭据，生产环境必须先确认访问边界。

## CORS 配置

本地开发不配置 CORS 时，后端默认允许任意 Origin，方便 Vite 代理和浏览器直接调试。生产或公网环境建议显式配置 `API_CORS_ORIGINS` 或 `api_cors_origins`，只填写前端控制台域名白名单。

预检请求当前支持 `GET,POST,PUT,PATCH,DELETE,OPTIONS`。如果新增需要浏览器调用的管理接口，同时使用了新的 HTTP 方法，需要同步更新后端 CORS 响应和这份文档。

## 最小云账号配置

没有真实云账号时可以先不配。需要查看真实云资源时，再配置这些变量：

```env
ALIYUN_ACCESS_KEY_ID=your-ak
ALIYUN_ACCESS_KEY_SECRET=your-secret

HUAWEI_ACCESS_KEY_ID=your-ak
HUAWEI_ACCESS_KEY_SECRET=your-secret
```

控制台新增云账号时不需要配置地域；后端会自动发现并采集全部可见地域。`ALIYUN_REGION`、`ALIYUN_REGIONS`、`HUAWEI_REGION`、`HUAWEI_REGIONS` 只保留给环境变量单账号兜底或接口调试限定范围。

K8s 中用 Secret 注入：

```bash
kubectl -n cloudlens create secret generic cloudlens-cloud-credentials \
  --from-literal=ALIYUN_ACCESS_KEY_ID=your-ak \
  --from-literal=ALIYUN_ACCESS_KEY_SECRET=your-secret \
  --from-literal=HUAWEI_ACCESS_KEY_ID=your-ak \
  --from-literal=HUAWEI_ACCESS_KEY_SECRET=your-secret
```

不要把真实密钥写进仓库。

## 当前架构取舍

- 后端默认使用 SQLite，所以 K8s 后端保持单副本。
- 暂时不引入 MySQL、Redis、Ingress、TLS、Helm 和后端多副本高可用。
- 前端只负责控制台展示，后端统一提供 `/api` 接口。
- 当前重点是资源可见性、监控可用性、可维护性和安全性。

## 简历可写点

- 使用 Kubernetes Deployment、Service、ConfigMap、Secret、PVC 部署前后端服务。
- 为后端配置 `/api/health` 存活和就绪探针，为 `/metrics` 预留 Prometheus 抓取入口。
- 使用 PVC 持久化 SQLite 数据、日志和运行态配置，说明了单副本与数据一致性的取舍。
- 使用 ServiceAccount 和 NetworkPolicy 做最小权限与东西向访问收敛。
- 基于 `client-go` 接入 kind 集群，提供 Node、Pod、Deployment 和 Warning Event 的只读巡检接口。
- 提供 Kustomize 拆分清单和 `all-in-one.yaml` 快速部署入口，兼顾学习体验和维护性。
