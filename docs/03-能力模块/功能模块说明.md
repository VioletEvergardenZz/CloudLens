# 功能模块

## 主线

- 多云资源监控
- 统一控制台

## 辅助

- 告警事件：只服务资源异常发现和处置闭环
- AI 摘要：只服务异常证据摘要和辅助判断
- 知识库：只沉淀可复用 SOP、处理记录和复盘结论
- 文件入云：只作为异常证据归档能力保留，不恢复为产品主线

## 当前已落地

- 阿里云账号保存和校验
- 阿里云 ECS/RDS 列表和监控概览
- 华为云账号保存、校验、ECS 列表和 CES 监控概览
- 多云风格控制台页面
- 统一资源索引接口：基于 ECS/RDS 快照输出 `/api/resources`
- 统一资源指标接口：通过 `/api/resources/{id}/metrics` 按资源 ID 查询 ECS/RDS 指标
- 统一巡检风险接口：通过 `/api/inspection/risks` 基于规范化资源索引识别风险
- Kubernetes 只读巡检：基于 kind 或 kubeconfig 查询 Node、Pod、Deployment 和 Warning Event
- K8s Node 与云主机关联：基于 Node 标签、`providerID`、`InternalIP` 和主机名匹配 ECS 资源

## 后续融合建议

- 文件入云不要再单独讲成产品主线，后续只作为“实例相关日志采样/归档”能力存在
- AI 不再单独讲成日志工具，后续只围绕实例异常做摘要和辅助诊断
- 知识库不再独立扩张，后续只沉淀云服务器排障经验和复盘结论
- K8s 不做完整管理平台，只作为“云资源到容器运行态”的只读巡检补充，后续重点验证 Node 与云主机的关联模型

## 其他功能重设计原则

当前阶段不再把“告警、AI、知识、同步任务、系统设置”做成并列产品模块，而是按主线重新收口：

- 告警事件改成“异常入口”：从资源、指标、同步失败中生成待确认事件，重点展示级别、影响资源、当前状态和下一步动作。
- 告警与处置改成“事件闭环”：只保留确认、处理中、已恢复、处理记录、恢复时间和复盘入口。
- AI 摘要改成“辅助判断”：输入只来自告警、指标、资源状态和采样新鲜度，输出只给影响判断、可能原因和检查建议。
- 知识沉淀改成“处置知识”：条目来源必须是告警结案、资源治理或故障复盘，命中后回流到资源详情和事件详情。
- 同步任务改成“同步健康”：解释资源为什么没出现、指标为什么不新鲜，账号级失败和单资源失败必须分开展示。
- 系统设置改成“接入边界”：只展示云账号、同步策略、安全边界和当前阶段启用能力，不做宽泛配置中心。

这几个入口的验收标准不是“页面是否存在”，而是是否能回答三个问题：

1. 当前哪个资源有风险？
2. 为什么判断它有风险？
3. 我下一步应该怎么查、怎么处置、怎么沉淀？
